# HookLine

HookLineは、テキストを入力して「ファイルへの追記」や「外部コマンドの実行」を瞬時に行うための、常駐型生産性向上ツールです。WSLでの堅牢なロジック開発と、Windows上での軽快なデスクトップ体験を両立させています。

## 開発方針
- **シンプル第一 (Simplicity First):** コードの単純さと可読性を最優先する。冗長なイベントリスナーや複雑な状態管理は避け、目的が達成できるなら最もシンプルな実装を選択する。必要であれば、既存機能を再設計してでもコードを簡潔に保つ。
- **ドキュメント・コメント管理 (Documentation & Comments):**
  - 非自明なロジック、ワークアラウンド、特殊な制御フロー（キャプチャフェーズの利用やOS固有の挙動など）には、必ずコード内にコメントを記述し、その意図と理由を残すこと。
  - コード修正時にこれらの既存コメントを削除したり、見落として上書きしないこと。
  - **仕様変更の同期:** アーキテクチャの変更や新しい仕様の導入を伴う修正を行う際は、ユーザーからの指示を待たず、エージェントが自律的に該当箇所の `GEMINI.md` を更新すること。仕様と実装の乖離を防ぐための必須作業とする。
  - **デバッグコード管理:** 動作確認のために一時的に追加したデバッグコード（ログ出力や一時的なガード等）には、必ず `// 動作確認用` というコメントを添えること。また、改修が一息ついたタイミングで、これらのコードは自律的に整理・削除すること。
- **漸進的変更 (Incremental Changes):** コードの書き換えは常に目的のために最小限かつ外科的（Surgical）に行う。書き換え目的以外の変更（リファクタリング、未関連の修正など）を同時に行わないこと。
- **機能的完全性 (Functional Integrity):** リファクタリングやシンプル化の際、既存の仕様、振る舞い、画面構成、UXは変更しないこと。機能のシンプル化と「仕様の簡素化（機能の削除）」は明確に区別すること。
- **構造的制約 (Structural Constraint):** バックエンドやロジックの構造を調整する際も、既存のUI/UX、操作性、ユーザーインターフェースの振る舞いと相反するような調整は行わないこと。
- **ドキュメント管理方針:** 本ドキュメントへの修正は原則として「追記」のみとし、情報の削除は行わない。未実装の機能や計画は削除せず、常に現状を反映した表現へ調整することで、経緯と意図を保持する。
- **分業体制:** コードの作成やファイルの修正は Gemini CLI が担当し、その内容の確認やコミット・プッシュ、ビルド、その他システム実行権限を伴う操作はユーザーが自身のターミナルで実行する。

## 確定仕様

### 1. コア機能

- **ファイル追記:** 設定した file_path の末尾に、入力したテキストを改行付きで追記します。
- **外部コマンド実行:** 入力テキストを $TEXT または {{text}} という変数として扱い、任意のコマンドをシェル経由で実行します。
  - Windows: cmd /C 経由で実行
  - Linux: sh -c 経由で実行
- **マルチプロファイル:** 最大9つの設定スロットを持ち、用途に応じて保存・切り替えが可能です。
- **送信履歴:** 送信済みの内容を保存し、履歴パネルからクリックすることで入力欄へ呼び戻せます。

### 2. ウィンドウ挙動 & UX

- **グローバル表示トグル:** Alt + Space でウィンドウの表示/非表示を切り替えます。
- **即時非表示:** 
  - Esc キーまたは送信完了時に、即座にウィンドウを隠します。
  - ウィンドウ外の透明な背景部分をクリックした場合も、即座にウィンドウを隠します（Backdrop click）。
- **自動非表示 (Auto-hide on focus loss):** 
  - 別アプリをクリックするなど、HookLine からフォーカスが外れた場合に即座にウィンドウを隠します。
  - カード内部でのクリックやドラッグ操作による意図しない非表示を防ぐため、キャプチャフェーズでのイベント判定を実装しています。
- **ヒットボックスの動的最適化 (Hit-box optimization):**
  - メニュー（プロファイル/履歴）が閉じている時は、ウィンドウサイズをカード領域のみに縮小し、カード外へのクリックが背後のアプリに透過されるようにします。
  - メニュー展開時はウィンドウを拡張し、カード外（透明領域）のクリックを検知してウィンドウを非表示にします。
- **プラットフォーム適応型外観 (Platform-aware Aesthetics):**
  - **Windows:** 透明ウィンドウで発生しやすい境界線の問題を回避するため、OS標準の影を無効化し、CSSで深みのある影を補完します。
  - **macOS:** ネイティブの影を活かし、CSSは控えめなアクセントとして調整します。
- **オートフォーカス & 全選択:** ウィンドウが表示されるたびに、入力欄へフォーカスし、既存のテキストを全選択します。
- **ウィンドウ位置の記憶 (Persistence):**
  - ユーザーが移動させたウィンドウの座標を記憶し、次回表示時も同じ位置に表示します。
- **ウィンドウの表示サイズと移動:**
  - 画面全体ではなく、カード領域のみを表示するコンパクトなサイズにします。
  - カードのヘッダーまたはカード自体をドラッグして移動可能です。
- **タスクバー表示の動的制御:** 
  - メインWindowが表示されている時のみタスクバーにアイコンを表示します（非表示時はトレイ常駐）。
- **非同期処理:** Enter 送信後の処理をバックグラウンドスレッドへオフロードし、UIの残像やフリーズを防ぎます。
- **入力欄の動的伸縮:** 
  - 初期表示は1行。
  - 入力に応じて入力欄およびウィンドウ全体が縦に伸縮する。
  - 最大 `MAX_LINES`（5行）まで伸縮し、それ以上は入力欄内部でスクロールバーを表示。
  - フッターは常に表示。

- **エラー表示:** バックエンド操作でエラーが発生した際は、オーバーレイを表示せず、フッター領域（ショートカットキー表示エリア）をエラー用レイアウトへ動的に切り替える。フッターは背景色をローズ色に変更し、エラー内容の表示とコピーボタン、閉じるボタンを提供することで、視認性と操作性を両立する。

## 開発・テスト環境

- **ロジックの分離とテスト:** logic.rs にコマンド文字列の置換ロジックを分離し、WSL上で cargo test を用いたユニットテストを可能にしています。
- **同期ワークフロー:** sync.sh により、WSL側でのテストにパスした場合のみ、Windows側の作業ディレクトリへソースコードを同期（rsync）します。
- **タスク管理規定 (Task.md):**
  - **完了定義:** 全てのタスクは、実装後にユーザーが動作を確認し、「OK」の承諾を得るまで完了（[x]）にしてはならない。
  - **自律的更新:** 開発過程で発見されたバグ、改善点、不足機能については、ユーザーの明示的な指示を待たずに、エージェントが自律的に `Task.md` へ項目を追加すること。
  - **進捗同期:** 修正の提案時、および修正完了時には必ず `Task.md` のステータスを最新の状態に保つこと。


## 操作方法

| キー操作 | アクション |
| :--- | :--- |
| Alt + Space | ウィンドウの表示 / 非表示 |
| Ctrl + 1 〜 9 | プロファイルの切り替え |
| Enter | 送信 (追記 & コマンド実行) |
| Shift + Enter | 入力欄内での改行 |
| Esc | ウィンドウを隠す |

## 開発サイクル

1. **コード修正:** WSL側で `src-tauri/src/logic.rs` などを編集。
2. **テスト & 同期:** `./sync.sh` を実行。テスト失敗時は同期が中断されます。
3. **プレビュー:** Windows側のターミナルで `bun tauri dev` を実行して動作を確認。

## 技術スタック

- **Backend:** Rust (Tauri v2)
- **Frontend:** Vue.js 3 (Composition API / `<script setup>`) + Vite

## アーキテクチャ・設計決定

- **置換ロジックの実装場所:** テキストの置換およびOS固有のコマンド実行ロジックは、Vue.js（フロントエンド）ではなく、Rust（バックエンド/src-tauri/src/logic.rs）に実装する。

- **設定ファイルの管理:** 開発環境（DEV）と本番環境で設定ファイルを分離して管理する。これにより、テスト実行時や開発中に本番の設定を破壊することなく、安全な切り替えを実現する。

- **バックエンドの実装方針 (Task 3):**
    - **ファイル追記:** `std::fs::OpenOptions` を使用し、安全な追記モードを実装する。パスの解決はホームディレクトリを基準に行う。
    - **外部コマンド実行:** `std::process::Command` を使用する。OS判定により Windows(`cmd /C`) / Linux(`sh -c`) を自動適用する。実行前に `logic.rs` の置換ロジックを経由させることでコマンドインジェクションを抑制する。
    - **実装フェーズ:** ファイル追記機能とコマンド実行機能は分離して実装する。
    - **エラーハンドリング:** バックエンド操作（ファイル追記、コマンド実行）はそれぞれ独立して try-catch でラップする。ファイル追記が失敗した場合はコマンド実行はスキップし、いずれの操作も失敗した際は、どちらのステップで失敗したか明確に識別できるエラーログを出力する。エラー通知にはネイティブの alert を避け、HookLine の UI スタイルと統一したオーバーレイコンポーネントを使用し、ユーザー体験を阻害しない設計とする。

- **CI/CD パイプライン:** GitHub Actions を使用し、メインブランチへのプッシュ時に自動ビルドを行う。さらに `v*` 形式のタグがプッシュされた際には、自動的に GitHub Releases を作成し、各 OS（Windows, macOS, Ubuntu）用のバイナリを添付する。リリースライフサイクルを完全に自動化し、配布の手間を最小限に抑える。

- **プロファイル永続化の統合:** プロファイルデータは個別のファイルではなく、既存の `settings.json` に配列として統合する。保存・読み込みには `getBaseDir()` を使用し、環境（Dev/Prod）に応じたパスを動的に解決する。

- **ウィンドウサイズ最適化:** ウィンドウサイズはハードコードせず、コンテンツ（メニュー展開時など）に応じて動的に計算する。展開時は固定の展開サイズ、閉鎖時はコンパクトなサイズを適用し、CSS の `overscroll-behavior: none` と組み合わせて、ガタつきのない滑らかなUXを実現する。

- **永続化の実装方針 (Task 3 & 5):**
    - **送信履歴:** `hookline_history.json` を使用し、アプリ起動時に `appConfigDir` から読み込む。データ構造は履歴リスト（ID, Profile名, Text, UNIXTIME）で統一し、古い文字列型タイムスタンプも許容する柔軟なローダーを実装する。
    - **プロファイル永続化:** ユーザー設定やプロファイルデータは、履歴と同様に永続化対象とし、アプリ状態の一部として管理する。
    - **入力欄UX:** ウィンドウ表示時に、前回の入力内容と一致する場合はテキストを全選択し、それ以外はキャレットを末尾に配置する。選択色はライトモード・ダークモードで視認性を最適化する。
